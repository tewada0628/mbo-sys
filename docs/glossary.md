## 1. ドメイン用語

### 評価サイクル

**英語**: Evaluation Cycle  
**定義**: NEWONE の MBO 制度における年次の評価周期。10月1日に始まり翌年9月30日に終わる1年間。「評価期」がシステム上の管理単位（レコード）であるのに対し、「評価サイクル」は制度上の周期概念を指す。  
**関連用語**: 評価期、フェーズ

---

### 評価期

**英語**: Evaluation Period  
**定義**: MBO評価サイクルをシステム上で管理するための単位レコード。10月1日〜翌9月30日の1年間に対応し、`periods` テーブルに1レコードとして格納される。フェーズ期間はこのレコードに紐づく。  
**コード上の表記**: `Period`（モデル名）/ `periods`（テーブル名）/ `periodId`（変数名）  
**関連用語**: 評価サイクル、フェーズ

---

### フェーズ

**英語**: Phase  
**定義**: 評価期内の時間的区分。以下の6種類がある。

| フェーズ名 | 略称 | 期間（標準） | 主な操作 |
| ------- | -- | ------- | ------ |
| 目標設定フェーズ | `GOAL_SETTING` | 10月（1か月） | 目標設定・承認申請 |
| 中間振り返りフェーズ | `MIDTERM` | 2月 | 中間振り返り入力・目標修正申請 |
| 360度評価フェーズ | `DEGREE_360` | 7月 | 360度スコアのインポート（HR/ADMIN操作） |
| 自己評価フェーズ | `SELF_REVIEW` | 8〜9月 | 自己評価入力 |
| 上長評価フェーズ | `MANAGER_REVIEW` | 8〜9月（自己評価と並行） | 上長評価入力 |
| 評価調整・確定フェーズ | `EVALUATION` | 10月 | 評価確定（HR/ADMIN） |

**コード上の表記**: `Phase`（型名）/ `phases`（カラム名）  
**関連用語**: 評価期

---

### 目標セット

**英語**: Goal Set  
**定義**: 1名の社員が1評価期に持つ目標のまとまり。KPI連動目標2件 + 組織貢献目標1件の計3件を内包し、承認フローと評価はこの単位で管理される。  
**コード上の表記**: `GoalSet`（モデル名）/ `goal_sets`（テーブル名）/ `goalSetId`（変数名）  
**関連用語**: 目標項目、承認フロー、MBO対象フラグ

---

### 目標項目

**英語**: Goal Item  
**定義**: 目標セット内の個々の目標。各目標項目には目標タイプ・連動パターン・達成基準（1.2/1.0/0.8水準）・ウェイト・公開範囲が設定される。  
**コード上の表記**: `Goal`（モデル名）/ `goals`（テーブル名）/ `goalId`（変数名）  

> **注意**: モデル名は `GoalItem` ではなく `Goal`。テーブル名も `goal_items` ではなく `goals`。

**関連用語**: KPI連動目標、組織貢献目標、ウェイト、目標タイプ

---

### 目標タイプ

**英語**: Goal Type  
**定義**: 目標項目の種別。以下の3種類がある。

| コード値 | 名称 | 件数制約 |
| ------ | -- | ------ |
| `KPI_1` | KPI連動目標（1本目） | 1件 |
| `KPI_2` | KPI連動目標（2本目） | 1件 |
| `ORG_CONTRIBUTION` | 組織貢献目標 | 1件 |

**コード上の表記**: `type`（`goals.type` カラムの値）  
**関連用語**: KPI連動目標、組織貢献目標

---

### KPI連動目標

**英語**: KPI-linked Goal  
**定義**: 会社・部署のKPIに直結する目標。1評価期に2件設定する（`KPI_1`・`KPI_2`）。連動パターン（分解・先行指標・役割設定）でKPIとの接続方法を定義する。  
**コード上の表記**: `type: 'KPI_1'` または `type: 'KPI_2'`（`goals.type` カラムの値）  
**関連用語**: 目標タイプ、連動パターン

---

### 組織貢献目標

**英語**: Organizational Contribution Goal  
**定義**: 部署横断の協力・チーム育成・組織力強化に関する目標。1評価期に1件設定する（`ORG_CONTRIBUTION`）。連動パターン（上位目標・チーム力向上）で貢献の方向性を定義する。  
**コード上の表記**: `type: 'ORG_CONTRIBUTION'`（`goals.type` カラムの値）  
**関連用語**: 目標タイプ、連動パターン

---

### 公開範囲

**英語**: Visibility  
**定義**: 目標項目ごとに設定する閲覧可能な対象の範囲。デフォルトは `DEPARTMENT`（部署内公開）。

| コード値 | 閲覧できる対象 |
| ------ | ----------- |
| `SELF_ONLY` | 本人と上長・HR・ADMINのみ |
| `DEPARTMENT` | 同一部署の全メンバー（デフォルト） |
| `COMPANY` | 全社員 |

**コード上の表記**: `visibility`（`goals.visibility` カラムの値）  
**関連用語**: 目標項目

---

### 承認フロー

**英語**: Approval Flow  
**定義**: 目標セットを正式確定させるための4段階のプロセス全体。「本人入力・申請 → 上長承認 → 事業部長承認 → 経営承認」の順で進む。フロー自体はプロセスの概念であり、各段階の申請は `approval_requests` テーブルの個別レコードとして管理される。  
**コード上の表記**: プロセス概念のため単一のモデルは存在しない。個々の承認申請は `ApprovalRequest`（モデル名）/ `approval_requests`（テーブル名）  
**関連用語**: 承認申請、承認ステップ、経営

---

### 承認申請

**英語**: Approval Request  
**定義**: 承認フロー内で各承認ステップごとに生成される個別のレコード。1回の申請につき承認ステップ数分のレコードが生成される。  
**コード上の表記**: `ApprovalRequest`（モデル名）/ `approval_requests`（テーブル名）  
**関連用語**: 承認フロー、承認ステップ

---

### 承認ステップ

**英語**: Approval Step  
**定義**: 4段階承認フローのどの段階にあるかを示す番号（1〜4）。1=上長、2=事業部長、3=経営に対応する。`approval_requests.step` カラムに格納される。

| ステップ値 | 承認者 | 遷移後のステータス |
| -------- | ---- | ------------- |
| 1 | 上長（直属） | `PENDING_DIVISION` |
| 2 | 事業部長（`DEPT_MANAGER`以上） | `PENDING_EXECUTIVE` |
| 3 | 経営（`EXECUTIVE` / HR / ADMIN） | `APPROVED` |

**コード上の表記**: `step`（`approval_requests.step` カラム）  
**関連用語**: 承認フロー、経営

---

### 経営

**定義**: 承認フローの第4段階（最終承認）を担う承認者の総称。システム上は `position = EXECUTIVE`（経営層）の社員、または `role = HR` / `role = ADMIN` のユーザーが操作できる。  
**コード上の表記**: `position: 'EXECUTIVE'`（`employees.position` カラムの値）  
**関連用語**: 承認フロー、承認ステップ、position

---

### 修正申請

**英語**: Revision Request  
**定義**: `APPROVED`（承認済み）状態の目標セットに対して、社員が内容変更を申請すること。変更後も同じ4段階の承認フローを経る。申請中も `goal_sets.status` は `APPROVED` を維持し続ける。承認されると新バージョンの目標として保存され、変更前の内容は履歴として保持される。  
**コード上の表記**: `request_type: 'GOAL_REVISION'`（`approval_requests.request_type` カラムの値）  
**関連用語**: 修正理由コード、上長修正依頼、再申請

---

### 上長修正依頼

**英語**: Manager Revision Request  
**定義**: 中間振り返り時に上長が社員に目標の修正を求める、上長起点の修正フロー。上長が `midterm_reviews.revision_requested = TRUE` を立てると社員に通知が届き、社員がF-06の修正申請を起票する。修正申請のフロー自体は社員起点と同じ（`GOAL_REVISION`）。  
**コード上の表記**: `revision_requested`（`midterm_reviews.revision_requested` カラム）  
**関連用語**: 修正申請、中間振り返り

---

### 修正理由コード

**英語**: Revision Reason Code  
**定義**: 修正申請時に選択する変更理由の区分。以下の6種類がある。

| コード | 内容 |
| --- | -- |
| `KPI_CHANGE` | 会社全体または部署KPIの方針変更 |
| `STANDARD_DEVIATION` | 前例のない取り組みで達成基準が大きく乖離した |
| `ROLE_CHANGE` | 周囲の退職・休職により役割が大きく変更された |
| `MIDTERM_ENTRY` | 中途入社・復職に伴う期中目標設定 |
| `EARLY_CLOSURE` | 休職・退職に伴う期中クローズ（評価の早期実施） |
| `GRADE_PROMOTION` | 中間振り返り時の等級2→3昇格に伴うMBO目標新規設定 |

**コード上の表記**: `revision_reason`（`goals.revision_reason` カラム）  
**関連用語**: 修正申請

---

### 差し戻し

**英語**: Rejection  
**定義**: 承認フロー中にいずれかの承認者が申請を却下すること。差し戻し後は `goal_sets.status = REJECTED` となる。社員は `REJECTED` 状態のまま目標を修正して再申請（→ `PENDING_MANAGER`）できる。`APPROVED`後の差し戻しは「最終承認後差し戻し」として別に扱う。  
**コード上の表記**: `status: 'REJECTED'`（`goal_sets.status` の値）  
**関連用語**: 承認フロー、再申請、最終承認後差し戻し

---

### 最終承認後差し戻し

**英語**: Post-Approval Rejection  
**定義**: `APPROVED`状態の目標セットを、`DEPT_MANAGER`以上の役職者が難易度調整のために差し戻す操作（旧称: 目標設定会議差し戻し）。通常の承認フロー中の差し戻し（`REJECTED`）とは異なる別ステータス（`MEETING_REJECTED`）として管理される。

**ステータス遷移:**

```text
APPROVED
  ↓ DEPT_MANAGER 以上が「最終承認後差し戻し」を実行（コメント必須）
MEETING_REJECTED
  ↓ 社員が目標を修正して再申請
PENDING_MANAGER
  ↓ 以降は通常の4段階承認フローを再実施
APPROVED
```

**コード上の表記**: `status: 'MEETING_REJECTED'`（`goal_sets.status` の値）  
**操作権限**: `position = DEPT_MANAGER`（部長・事業部長）以上、および `HR`・`ADMIN`。`UNIT_MANAGER`（ユニット長）は不可。  
**関連用語**: 差し戻し、再申請

---

### 再申請

**英語**: Re-submission  
**定義**: 差し戻し（`REJECTED` または `MEETING_REJECTED`）を受けた社員が、目標を修正して再度承認申請を行うこと。再申請すると `goal_sets.status` は `PENDING_MANAGER` に遷移し、4段階承認フローを最初から再実施する。  
**コード上の表記**: `POST /api/goals/:goalSetId/submit`（再申請も同一エンドポイントを使用）  
**関連用語**: 差し戻し、最終承認後差し戻し、承認フロー

---

### 中間振り返り

**英語**: Midterm Review  
**定義**: 2月に実施する、上期の進捗確認と目標調整の場。社員コメントと上長振り返りコメントを各目標項目ごとに並列入力する。  
**コード上の表記**: `MidtermReview`（モデル名）/ `midterm_reviews`（テーブル名）  
**関連用語**: 中間振り返りの完了、上長振り返りコメント、上長修正依頼

---

### 中間振り返りの完了

**定義**: 中間振り返りが「提出完了」とみなされる条件。`midterm_reviews.employee_submitted_at`（社員提出日時）と `midterm_reviews.manager_submitted_at`（上長提出日時）の両カラムに値が入った時点で完了となる。どちらが先に提出しても構わない。  
**コード上の表記**: `employee_submitted_at` / `manager_submitted_at`（`midterm_reviews` カラム）  
**関連用語**: 中間振り返り

---

### 上長振り返りコメント

**英語**: Manager Midterm Comment  
**定義**: 中間振り返り時に上長（1次評価者）が各目標項目に対して入力するコメント。社員コメントと画面上で並列表示される。上長評価フェーズの「上長評価コメント」とは別物。  
**コード上の表記**: `manager_comment`（`midterm_reviews.manager_comment` カラム）  
**関連用語**: 中間振り返り、上長評価

---

### 自己評価

**英語**: Self Review  
**定義**: 評価期末（8〜9月）に社員が各目標項目の達成度を評価すること。各目標に評価スコアとコメントを入力する。自己評価を提出すると上長評価が入力可能になる。  
**コード上の表記**: `SelfReview`（モデル名）/ `self_reviews`（テーブル名）  
**関連用語**: 評価スコア、上長評価

---

### 上長評価

**英語**: Manager Review  
**定義**: 社員の自己評価を受けて、上長が各目標項目に評価スコアとコメントを入力する評価。自己評価提出後にのみ入力可能。自己評価との差異がある場合はコメント記入が必須。入力画面にはバイアス警告が表示される。  
**コード上の表記**: `ManagerReview`（モデル名）/ `manager_reviews`（テーブル名）  
**関連用語**: 自己評価、評価スコア、バイアス警告

---

### 360度評価

**英語**: 360-Degree Evaluation  
**定義**: 上長・同僚・部下など複数の視点から受ける評価。7月に本システムとは別のサーベイツールで実施し、結果を CSV でインポートする。「成果スコア」と「クレドスコア」の2軸があり、MBOスコアへの加算に使用される（詳細な加算ロジックは `docs/specification.md` F-10 参照）。  
**コード上の表記**: `Degree360Score`（モデル名）/ `degree_360_scores`（テーブル名）  
**関連用語**: MBOスコア、評価調整・確定

---

### 評価調整・確定

**英語**: Evaluation Adjustment / Final Evaluation  
**定義**: 人事担当者（HR/ADMIN）が全社員分の評価スコアを確認・調整し、最終スコアを確定させる作業。MBOスコアと360度スコアを合算して最終評価スコアを算出し、総合評価（S/A/B/C/D）を確定する。  
**コード上の表記**: `FinalEvaluation`（モデル名）/ `final_evaluations`（テーブル名）  
**関連用語**: MBOスコア、360度評価、総合評価

---

### ウェイト

**英語**: Weight  
**定義**: 各目標項目の評価への貢献割合（%）。3件の目標項目のウェイト合計は必ず100%になる。デフォルト値は KPI_1=50% / KPI_2=30% / ORG_CONTRIBUTION=20%。APIバリデーションとDBトリガーの両方で合計100%を強制する。  
**コード上の表記**: `weight`（`goals.weight` カラム）  
**関連用語**: 目標項目、MBOスコア

---

### MBO対象フラグ

**英語**: MBO Target Flag  
**定義**: 当該目標セットが承認フロー・MBO評価の正式対象であるかを示すフラグ。`employee_type = REGULAR` かつ `grade >= 3` の社員の目標セットで `TRUE` となる。等級1〜2・契約社員・アシスタントは目標入力は可能だが `FALSE`。  
**コード上の表記**: `is_mbo_target`（`goal_sets.is_mbo_target` カラム）  
**関連用語**: 等級、雇用形態

---

### 評価対象外フラグ

**英語**: Evaluation Exempt Flag  
**定義**: 評価対象期間における実勤務期間が6か月未満の場合に立てるフラグ。フラグが立つと評価スコアの算出や昇降給判定から除外される。目標設定自体は可能（文化・共有目的）。  
**コード上の表記**: `is_evaluation_exempt`（`employees.is_evaluation_exempt` カラム）  
**関連用語**: イレギュラー対応、対象月数

---

### 有効フラグ

**英語**: Active Flag  
**定義**: 同一社員・同一評価期で複数の goal_set レコードが存在するとき（期中昇格時の旧 goal_set 論理削除等）、有効なレコードを1件に限定するためのフラグ。`TRUE` のレコードのみが現在有効な目標セット。  
**コード上の表記**: `is_active`（`goal_sets.is_active` カラム）  
**関連用語**: 論理削除、期中昇格

---

### 論理削除

**定義**: レコードを物理的に削除するのではなく、有効フラグ（`is_active`）を `FALSE` にすることで削除扱いとするパターン。期中昇格時に既存の goal_set を無効化する際に使用する。削除後も履歴として参照可能。  
**コード上の表記**: `is_active = false`（`goal_sets` テーブルで使用）  
**関連用語**: 有効フラグ、期中昇格

---

### 対象月数

**英語**: Target Months  
**定義**: 休職・中途入社・期中昇格等のイレギュラーケースで、社員が当期にMBO対象として在籍した月数。昇給額の按分計算等に使用される。  
**コード上の表記**: `target_months`（`goal_sets.target_months` カラム）  
**関連用語**: イレギュラー対応、評価対象外フラグ

---

### イレギュラー対応

**英語**: Irregular Case Handling  
**定義**: 通常の評価サイクルから外れるケース（休職・復職・中途入社・退職・期中昇格）への対応の総称。ケースごとの処理方針は以下の通り。

| ケース | 処理方針 |
| ---- | ------ |
| 中途入社・復職 | 入社/復職タイミングで期中目標設定を許可（修正理由: `MIDTERM_ENTRY`） |
| 休職予定・退職 | 期末を待たず期中クローズを実施（修正理由: `EARLY_CLOSURE`） |
| 等級2→3昇格（2月） | 旧 goal_set を論理削除し、新 goal_set（`is_mbo_target = TRUE`）を作成して承認フロー開始（修正理由: `GRADE_PROMOTION`） |
| 実勤務6か月未満 | 評価対象外フラグ（`is_evaluation_exempt = TRUE`）を立て、昇降給算定から除外 |

**関連用語**: 期中クローズ、期中昇格、評価対象外フラグ、対象月数

---

### 期中クローズ

**英語**: Early Closure  
**定義**: 評価期の途中で、休職予定者（実勤務6か月以上）や退職者がその時点までの自己評価・上長評価を完了させ、評価プロセスを早期終了させること。HR/ADMINが強制終了操作を行うことも可能。修正申請の修正理由コード `EARLY_CLOSURE` を使って処理する。  
**コード上の表記**: `revision_reason: 'EARLY_CLOSURE'`  
**関連用語**: イレギュラー対応、修正理由コード

---

### 期中昇格

**英語**: Mid-term Grade Promotion  
**定義**: 中間振り返り（2月）のタイミングで等級2から等級3に昇格すること。昇格時の処理: 既存の goal_set を論理削除（`is_active = FALSE`）し、`is_mbo_target = TRUE` の新 goal_set を作成して4段階承認フローを開始する。`target_months = 8`（2〜9月）を記録する。  
**コード上の表記**: `revision_reason: 'GRADE_PROMOTION'`  
**関連用語**: イレギュラー対応、論理削除、有効フラグ、MBO対象フラグ、対象月数

---

## 2. 評価制度用語

### 等級

**英語**: Grade  
**定義**: NEWONE の人事制度における職位区分。等級3以上の正社員が MBO 評価対象。等級1〜2の正社員および契約社員・アシスタントは MBO 評価対象外。等級7〜8相当が部長・事業部長（`DEPT_MANAGER`）、等級5〜6相当がユニット長（`UNIT_MANAGER`）に対応する。  
**コード上の表記**: `grade`（`employees.grade` カラム）  
**関連用語**: MBO対象フラグ、雇用形態

---

### 雇用形態

**英語**: Employee Type  
**定義**: MBO評価制度の適用ルールを決定する社員属性。ロール（`role`）とは独立した別軸。

| コード値 | 雇用形態 | MBO評価 | 360度評価（受ける側） |
| ------ | ------ | ----- | --------------- |
| `REGULAR` | 正社員 | 等級3以上で対象 | 対象 |
| `CONTRACT` | 契約社員 | 対象外 | 育成目的で対象 |
| `ASSISTANT` | アシスタント | 対象外 | 対象 |

**コード上の表記**: `employee_type`（`employees.employee_type` カラムの値）  
**関連用語**: 等級、MBO対象フラグ

---

### 連動パターン

**英語**: KPI Pattern  
**定義**: 目標項目が上位のKPI・組織目標とどう繋がるかを示すパターン。KPI連動目標と組織貢献目標でそれぞれ選択肢が異なる。

**KPI連動目標用:**

| コード値 | 名称 | 概要 |
| ------ | -- | -- |
| `KPI_DECOMPOSITION` | 分解 | チームや部署のKPIを個人目標へ分解する |
| `LEADING_INDICATOR` | 先行指標 | 最終KPIへつながる先行指標を設定する |
| `ROLE_IN_GOAL` | 役割設定 | チーム目標に対する自分の役割・貢献を定義する |

**組織貢献目標用:**

| コード値 | 名称 | 概要 |
| ------ | -- | -- |
| `UPPER_GOAL` | 上位目標 | 部門や上位組織のKPIをそのまま担う |
| `TEAM_GROWTH` | チーム力向上 | チームの育成・改善に貢献する目標 |

**コード上の表記**: `kpi_pattern`（`goals.kpi_pattern` カラム）  
**関連用語**: KPI連動目標、組織貢献目標

---

### 達成基準

**英語**: Achievement Criteria  
**定義**: 各目標項目に設定する3段階の達成レベル。各水準の定義は以下の通り。

| 水準 | 名称 | 定義 |
| -- | -- | -- |
| 1.2 | 挑戦目標 | 1.0の1.2倍以上で、組織の発展に影響するアウトカムが出せるレベル |
| 1.0 | 達成目標 | KPIと等級定義から今回目指すべきストレッチな目標。達成できるかギリギリのライン |
| 0.8 | 最低目標 | 1.0の約8割。着実に努力すれば達成可能なレベル |

**コード上の表記**: `criteria_1_2` / `criteria_1_0` / `criteria_0_8`（`goals` カラム）  
**関連用語**: 評価スコア、MBOスコア

---

### 評価スコア

**英語**: Evaluation Score / Goal Score  
**定義**: 自己評価・上長評価で各目標項目に設定するスコア値。達成基準の水準に対応する。MBOスコア（目標セット全体の加重平均）とは区別する。

| スコア値 | 意味 |
| ------ | -- |
| 1.2 | 達成基準1.2水準（挑戦目標）を達成 |
| 1.0 | 達成基準1.0水準（達成目標）を達成 |
| 0.8 | 達成基準0.8水準（最低目標）を達成 |
| 1.5 / 2.0 | 特例（特筆すべき成果）。上長評価のみ設定可能 |

**コード上の表記**: `score`（`self_reviews.score` / `manager_reviews.score` カラム）  
**関連用語**: 達成基準、MBOスコア

---

### MBOスコア

**英語**: MBO Score  
**定義**: 各目標項目の評価スコア × ウェイトの加重平均で算出される、目標セット全体のスコア。計算式: `Σ（goal[i].score × goal[i].weight / 100）`。360度スコアと合算して最終評価スコアを算出する。  
**コード上の表記**: `mbo_score`（`final_evaluations.mbo_score` カラム）  
**関連用語**: 評価スコア、ウェイト、360度評価、総合評価

---

### 総合評価

**英語**: Overall Evaluation  
**定義**: MBOスコアと360度スコアを合算した最終評価スコアに基づいてHR/ADMINが確定する5段階の評価結果（S/A/B/C/D）。評価調整・確定フェーズで確定する。

| 評価 | 割合目安 | 内容 |
| -- | ------ | -- |
| S | 5% | 特筆すべき成果 |
| A | 15% | 期待を超える成果 |
| B | 60% | 期待通りの評価（MBOスコア100以上） |
| C | 15% | 期待を下回る評価 |
| D | 5% | 現状の役割等を見直すレベル |

**コード上の表記**: `overall_grade`（`final_evaluations.overall_grade` カラム）  
**関連用語**: MBOスコア、評価調整・確定

---

### バイアス警告

**英語**: Bias Warning  
**定義**: 上長評価入力画面（S-07）の上部に表示される、評価バイアス五大症状のガイダンスバナー。特定のスコアパターンをシステムが自動検知するものではなく、上長に注意を促す静的なガイダンスとして表示される（F-18）。バナーは折りたたみ可能。

| 症状名 | 注意の内容 |
| ---- | -------- |
| ハロー効果 | 全体的な印象が個々の評価項目に影響していませんか？ |
| 寛大化傾向 | 思い入れや同情から評価が甘くなっていませんか？ |
| 厳格化傾向 | 優秀な誰かを基準に評価が厳しくなっていませんか？ |
| 中心化傾向 | 差をつけたくないために評価が中間に集まっていませんか？ |
| 期末効果 | 評価直前の印象だけで判断していませんか？ |

**コード上の表記**: `BiasWarningBanner`（コンポーネント名）  
**関連用語**: 上長評価

---

## 3. ユーザー・ロール用語

### role（権限区分）

システム上の操作権限を表す。1ユーザーが複数ロールを持つことができる（例: チームリーダーは `TEAM_LEADER` と `MEMBER` を同時に保持）。

| ロール値 | 日本語 | 主な権限 |
| ------ | ---- | ------ |
| `MEMBER` | 一般社員 | 自身の目標設定・評価入力・他者目標閲覧（公開範囲設定に従う） |
| `TEAM_LEADER` | チームリーダー | MEMBER権限 + 担当チームメンバーの目標閲覧・コメント入力 |
| `MANAGER` | 上長・管理職（ユニット長・部長・事業部長を含む） | TEAM_LEADER権限 + 配下メンバーの目標承認・上長評価入力 |
| `HR` | 人事担当者 | MANAGER権限 + 全社評価集計・調整・イレギュラー対応・マスタ管理 |
| `ADMIN` | システム管理者 | 全権限（組織・ユーザー・評価期マスタの管理） |

> **注意**: `role`（権限区分）と `position`（役職区分）は独立した別カラム。`MANAGER` ロールにはユニット長・部長・事業部長が含まれるが、最終承認後差し戻し権限は `position = DEPT_MANAGER` 以上に限定される。

---

### position（役職区分）

評価制度上の役職ランクを表す。権限（`role`）とは独立した別軸で、承認権限や操作権限の一部に影響する。

| 値 | 役職 | 等級相当 |
| -- | -- | ------ |
| `MEMBER` | 一般（役職なし） | 等級1〜4相当 |
| `TEAM_LEADER` | チームリーダー | 等級4〜5相当 |
| `UNIT_MANAGER` | ユニット長 | 等級5〜6相当 |
| `DEPT_MANAGER` | 部長・事業部長 | 等級7〜8相当 |
| `EXECUTIVE` | 経営層 | 等級9以上相当 |

---

### 上長

**英語**: Direct Manager  
**定義**: 組織ツリー上で社員の直属の上位者。承認フローにおいて第1ステップの承認者となる（`role = MANAGER`）。`employees.manager_id` で参照される。  
**コード上の表記**: `managerId`（変数名）/ `manager_id`（カラム名）  
**関連用語**: 承認フロー、承認ステップ

---

### 事業部長

**英語**: Division Manager  
**定義**: 事業部を統括する管理職。承認フローの第2ステップを担当する。`position = DEPT_MANAGER` で識別される。部長も同じ `DEPT_MANAGER` 値を使用する。  
**コード上の表記**: `position: 'DEPT_MANAGER'`（`employees.position` カラムの値）  
**関連用語**: 承認フロー、承認ステップ、最終承認後差し戻し

---

## 4. ステータス・種別用語

### goal_sets.status

| ステータス値 | 日本語ラベル | 説明 |
| ---------- | --------- | -- |
| `DRAFT` | 下書き | 社員が入力中。未申請または差し戻し（`REJECTED`）後に修正中 |
| `SAVED` | 保存済み | 等級1〜2・契約社員・アシスタントが目標を保存した状態。承認フローには入らない |
| `PENDING_MANAGER` | 上長承認待ち | 社員が申請済み。上長の承認待ち |
| `PENDING_DIVISION` | 事業部長承認待ち | 上長承認済み。事業部長の承認待ち |
| `PENDING_EXECUTIVE` | 経営承認待ち | 事業部長承認済み。経営の承認待ち |
| `APPROVED` | 承認済み | 全段階の承認完了。修正申請中もこのステータスを維持する |
| `REJECTED` | 差し戻し | 承認フロー中にいずれかの段階で差し戻された状態。社員が修正して再申請可能 |
| `MEETING_REJECTED` | 最終承認後差し戻し | `APPROVED`後に `DEPT_MANAGER` 以上が難易度調整のために差し戻した状態 |

> **注意**: `PENDING_APPROVAL` という値はDBには存在しない。UI上で `PENDING_*` 系ステータスをまとめて「承認待ち」と表示する際の画面表示用の総称として使うにとどめる。

### approval_requests.status

| ステータス値 | 日本語ラベル | 説明 |
| ---------- | --------- | -- |
| `PENDING` | 承認待ち | 現在の承認ステップで承認を待っている状態 |
| `APPROVED` | 承認済み | 当該ステップの承認が完了した状態 |
| `REJECTED` | 差し戻し | 当該ステップで差し戻された状態 |
| `REVISION_PENDING` | 修正申請中 | 修正申請が提出され、承認を待っている状態 |
| `REVISION_APPROVED` | 修正承認済み | 修正申請が承認された状態 |
| `REVISION_REJECTED` | 修正差し戻し | 修正申請が差し戻された状態 |

### approval_requests.request_type（申請種別）

| 種別値 | 日本語 | 説明 |
| ----- | ---- | -- |
| `INITIAL_APPROVAL` | 初回承認申請 | 目標設定後の初回承認フロー |
| `GOAL_REVISION` | 目標修正申請 | `APPROVED`後の修正申請（社員起点・上長起点どちらも同じ値） |
| `MEETING_REJECTION` | 最終承認後差し戻し | 部長以上による難易度調整差し戻し操作 |
