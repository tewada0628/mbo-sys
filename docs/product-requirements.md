# プロダクト要求定義書

**プロジェクト名**: 社内MBO管理システム  
**バージョン**: 1.0  
**作成日**: 2026-04-30  
**対象読者**: 開発担当者、プロジェクトマネージャー、人事担当者  
**フェーズ**: ベータ版（Vercel + Supabase 構成）

---

## 1. プロダクトビジョンと目的

### 1.1 ビジョン

社員一人ひとりの目標と貢献が組織全体から見える状態をつくり、「自分の頑張りが正当に評価されている」という実感を全社員が持てる組織を目指す。

そのために、Excel 分散管理による情報の孤立を解消し、目標・評価・履歴を一元管理する Web システムを構築する。

### 1.2 背景と課題

現状（社員数 約80名、将来目標 300名）、MBO目標の管理は Excel ファイルを SharePoint の部署ごとのフォルダで運用している。この運用には以下の課題がある。

| 課題 | 内容 | 影響規模 |
| --- | --- | ------ |
| 発見性の低さ | ファイルの格納場所がわかりにくく、他者の目標を参照するコストが高い | 全社員（約80名）が対象。参照のたびに担当者への問い合わせが発生 |
| 履歴の追いにくさ | 期ごとの目標・達成度の変遷を時系列で確認することが困難 | 評価期ごとに Excel が増殖し、過去データの参照に時間を要する |
| 協力関係の希薄化 | 他の社員の目標を知る機会が限られ、横断的な協力が生まれにくい | 部署をまたいだ目標連携が属人的なコミュニケーションに依存している |
| 目標修正管理の困難さ | 期中の修正履歴が残らず、評価時に経緯の確認ができない | 修正の根拠が口頭・メールに散在し、評価調整時に確認コストが発生 |

### 1.3 目的

本システムは以下の3点を達成することで、1.1 のビジョン実現に貢献する。

1. **運用コストの削減**: MBO管理を Excel から Web に移行し、人事担当者の集計・調整工数を削減する
2. **透明性の向上**: 社員が互いの目標を参照できる環境を整備し、部署横断の協力関係を促進する
3. **履歴の保全**: 組織変更・人事異動後も目標・評価・修正経緯を正確に追跡可能な状態を維持する

## 2. ターゲットユーザーと課題・ニーズ

### 2.1 ユーザー区分

対象社員数: 現在 約80名 / 将来目標 300名（同時接続数 最大100名）

#### 主要ユーザー（評価フローの主体）

| ユーザー区分 | 主な役割 | ペインポイント | 主な利用タイミング |
| ------ | ---- | ------- | --------- |
| 一般社員・等級3以上（MEMBER） | 自身の目標設定・自己評価・承認申請 | 承認がどこまで進んでいるかわからない。目標の書き方に迷う | 目標設定期（10月）・中間振り返り（2月）・評価期（8〜9月） |
| 上長・管理職（MANAGER） | 配下メンバーの目標承認・上長評価入力 | 複数メンバーの承認が重なる時期に処理が追いつかない。誰の承認が残っているか一目でわからない | 承認依頼があるたびに随時（評価期に集中） |
| チームリーダー（TEAM_LEADER） | 担当チームの目標確認・進捗コメント | メンバーの進捗を把握して適切なフォローができているか不安。自分のコメントがメンバーに届いているか確認しづらい | 中間振り返り（2月）を中心に随時 |
| 人事担当者（HR） | 全社員の評価集計・調整・イレギュラー対応 | 評価確定期に全社分の集計・調整が手作業で大きな負担。休職・退職者の個別対応がシステムに乗せにくい | 評価確定期（10月）が最繁忙。イレギュラー対応は随時 |
| システム管理者（ADMIN） | 組織・ユーザー・評価期マスタの管理 | 組織改編時のマスタ更新に漏れがないか不安。設定ミスが評価フローに波及することへの懸念 | 評価期開始前・組織変更時のみ（非日常的） |

## 4. 成功の定義

### 4.1 定量目標

| 指標 | 現状（ベースライン） | 目標値 | 計測方法 | 測定タイミング |
| -- | ---------- | --- | ---- | ------- |
| 目標設定率（等級3以上正社員） | 100% | 100% | `goal_sets.status = APPROVED` の件数 ÷ 評価対象者数 | 目標設定フェーズ終了時（12月末） |
| 目標閲覧率 | 計測不可（Excel運用のため） | 社員の80%以上が月1回以上閲覧 | `/goals` ページのユニーク訪問者数（要: アクセスログ実装） | 初回評価期終了後（翌年9月） |
| 評価完了率 | 100% | 評価対象者の100% | `self_reviews`・`manager_reviews` の完了レコード数 ÷ 評価対象者数 | 評価フェーズ終了時（9月末） |

### 4.2 定性目標

| 定性目標 | 検証方法 |
| ---- | ---- |
| 評価プロセスへの納得度が向上する | 評価期終了後のサーベイで満足度スコア 80点以上 |
| 人事担当者の集計工数が削減される | HR担当者へのヒアリングで「以前と比べて楽になった」と回答する割合 80%以上 |
| 組織横断の協力関係が生まれる | 定性評価（初回評価期後にHRがヒアリングで確認） |

> 成功指標ではなく、システム稼働開始の前提条件として定義する。

- リリース後の評価期（2026年10月〜）からExcelによる目標管理を廃止する
- 廃止確認: HR担当者がSharePointのMBOフォルダへの更新停止を確認する

## 5. ビジネス要件

### 5.1 評価サイクル

評価サイクルは年1回（10月開始・翌年9月終了）。

```
10月          2月                    7月              8〜9月          10月
 |             |                      |                 |               |
目標設定      中間振り返り              360度評価実施      自己評価         評価調整
              目標修正申請              昇降格判断         上長評価         最終評価確定
              360度サーベイ中間評価     登用/降職の優先実施
```

| フェーズ | 期間 | 主要アクション | 切り替え方法 |
| ---- | -- | ------- | ------ |
| 目標設定 | 10月1日〜10月31日 | MBO目標3件の設定・承認 | HR/ADMINが手動で切り替え |
| 中間振り返り | 2月1日〜2月28日 | 進捗確認・目標修正申請・360度サーベイ中間評価 | HR/ADMINが手動で切り替え |
| 360度評価 | 7月1日〜7月31日 | 360度評価実施・昇降格判断 | HR/ADMINが手動で切り替え |
| 自己評価・上長評価 | 8月1日〜9月30日 | MBO達成度の自己評価・上長評価 | HR/ADMINが手動で切り替え |
| 評価調整・確定 | 10月1日〜10月31日 | 最終評価の調整・確定 | HR/ADMINが手動で切り替え |

### 5.2 目標構成

1社員につき1評価期に対して1つの目標セット（3件）を作成する。

| 目標タイプ | デフォルト重み |
|-----------|-------------|
| KPI連動目標1（`KPI_1`） | 50% |
| KPI連動目標2（`KPI_2`） | 30% |
| 組織貢献目標（`ORG_CONTRIBUTION`） | 20% |

重みのデフォルト値は変更可能。ただし3目標の合計は必ず100%とする（APIバリデーションおよびDBトリガーで担保）。

### 5.3 承認フロー

目標設定および目標修正申請は4段階の承認フローを経て確定する。

1. 本人申請 → `PENDING_MANAGER`（直属上長承認待ち）
2. 直属上長承認 → `PENDING_DIVISION`（事業部長承認待ち）
3. 事業部長承認 → `PENDING_EXECUTIVE`（経営承認待ち）
4. 経営承認 → `APPROVED`（確定）

**共通ルール:**

- 各ステップの目安期限: 5営業日以内
- 承認者不在時の扱い: 代理承認者を設定できるようにする（代理承認の仕組みの有無は未定）
- 差し戻し時はコメント入力必須。差し戻し回数の上限なし
- 差し戻された場合、目標セットは `REJECTED` ステータスとなり、申請者が修正して再申請する

### 5.4 総合評価の分布目安（等級3以上）

| 評価 | 割合目安 | 昇給率目安 | 内容 |
| -- | ---- | ----- | -- |
| S | 5% | 5〜15% | ゼロ人でも良いくらい特筆すべき成果 |
| A | 15% | 3〜6% | 期待を超える成果 |
| B | 60% | 1〜2% | 期待通りの評価（MBOスコア100以上） |
| C | 15% | 0〜-5% | 期待を下回る評価 |
| D | 5% | -5〜-10% | 現状の役割等を見直すレベル |

### 5.5 昇降給ロジック（等級2以下）

等級2以下はMBO評価対象外のため、360度評価「成果」スコアのみで昇降給を決定する。

| 成果スコア | 昇降給 |
| ----- | --- |
| 3.0未満 | 降給対象（審議） |
| 3.0以上3.5未満 | -3% |
| 3.5以上4.5未満 | +3% |
| 4.5以上 | 部門長会議以上で+5%を検討 |

## 6. ユーザーストーリー

> 形式: 「〈ユーザー〉として、〈目的〉したい。なぜなら〈価値・理由〉だから。」

### 一般社員・等級3以上（MEMBER）

- 一般社員として、目標設定フェーズにKPI目標2件と組織貢献目標1件を設定して承認申請したい。なぜなら自分の目標を組織に公約することで、評価の根拠を明確にしたいから。
- 一般社員として、承認が現在どのステップにあるかをリアルタイムで確認したい。なぜなら承認が止まっていても気づけず、フェーズ期限を過ぎてしまうリスクがあるから。
- 一般社員として、他の社員の公開目標を閲覧したい。なぜなら部署横断の協力機会を自分から見つけられるようにしたいから。
- 一般社員として、中間振り返り時に各目標の進捗コメントを入力し、上長のフィードバックを受けたい。なぜなら期末前に軌道修正の機会を確保したいから。
- 一般社員として、自己評価スコアとコメントを入力したい。なぜなら自分の達成度を客観的に記録し、上長評価との差異を把握したいから。
- 一般社員として、過去の評価結果・目標を期別に閲覧したい。なぜなら自分の成長を振り返り、今期の目標設定の参考にしたいから。

### チームリーダー（TEAM_LEADER）

- チームリーダーとして、担当チームのメンバー全員の目標と進捗を一覧で確認したい。なぜなら遅れているメンバーを早期に把握し、適切なフォローをしたいから。
- チームリーダーとして、中間振り返り時に各メンバーへ進捗コメントを入力したい。なぜなら口頭だけでなく記録として残すことで、伝達漏れや認識のずれを防ぎたいから。

### 上長・管理職（MANAGER）

- 上長として、配下メンバーの承認申請を一覧で確認し、承認または差し戻しができたい。なぜなら複数メンバーの申請が重なる時期でも処理漏れをなくしたいから。
- 上長として、中間振り返りで修正が必要なメンバーに修正依頼フラグを立てたい。なぜなら口頭指示だけでは記録が残らず、後から経緯を確認できないから。
- 上長として、配下メンバーの上長評価スコアとコメントを入力し、自己評価との差異がある場合にコメントを残したい。なぜなら評価の根拠を透明化し、メンバーの納得度を高めたいから。
- 部長（DEPT_MANAGER）として、経営承認済みの目標を難易度調整のために差し戻したい。なぜなら全社の難易度水準を揃えるプロセスをシステム上で完結させ、記録として残したいから。

### 人事担当者（HR）

- 人事担当者として、全社員の目標・評価データを一覧で確認・集計したい。なぜなら評価確定期の集計作業をExcelなしで完結させ、手作業ミスをなくしたいから。
- 人事担当者として、MBOスコアと360度評価スコアを合算した最終評価を調整・確定したい。なぜなら評価ルールをシステムで一貫して適用し、全社の公平性を担保したいから。
- 人事担当者として、休職・復職・退職者の評価プロセスを期中でクローズまたは設定したい。なぜなら例外的なケースの処理をシステムに乗せ、個別対応のオペレーションを標準化したいから。
- 人事担当者として、SmartHRからエクスポートしたCSVで等級・雇用形態・評価者を一括更新したい。なぜなら人事異動のたびに手動でシステムを更新する工数を削減したいから。

### システム管理者（ADMIN）

- システム管理者として、組織マスタ（部署・組織構造）を管理したい。なぜなら組織変更時に評価フローが正しく機能し続けるようにしたいから。
- システム管理者として、評価期とフェーズ期間を設定・切り替えたい。なぜなら評価サイクルの開始・終了を確実に制御し、フェーズ外操作を防ぎたいから。

### 副次ユーザー

- 等級1〜2の社員として、目標を入力・保存して同僚に共有したい。なぜなら承認フロー対象外であっても、自分の取り組みを組織内で可視化したいから。
- 契約社員として、育成目的の目標を記録したい。なぜなら契約更新時の評価材料として、自分の成長を証明できる記録を残したいから。

## 7. 受け入れ条件

### 認証（F-01）

- [ ] 社員がメールアドレスを入力すると、8桁のワンタイムコード（OTP）がそのアドレス宛に届くこと
- [ ] OTPを入力してログインができること（コードの有効期限は60分）
- [ ] ログアウト後はセッションが即時無効化されること
- [ ] システムから削除（login_enabled=false）されたユーザーはログインできないこと

### 目標設定（F-05）

- [ ] 等級3以上の正社員が3件の目標を入力し、4段階の承認フローを経て `APPROVED` になること
- [ ] 等級1〜2の社員は目標入力・保存はできるが、承認申請ボタンが表示されないこと
- [ ] 承認済み・承認申請中の目標は社員が自由編集できず、APIが `403 Forbidden` を返すこと
- [ ] 3目標のウェイト合計が100%でない場合、APIバリデーションとDBトリガーの両方で弾かれること

### 目標修正申請（F-06）

- [ ] 修正理由の選択（6種類のコード）と記入が必須となること
- [ ] 修正申請中も `goal_sets.status` は `APPROVED` のまま維持されること
- [ ] 承認後に旧バージョンが履歴として保持されること

### 中間振り返り（F-07）

- [ ] 社員の提出（`employee_submitted_at`）と上長の提出（`manager_submitted_at`）の両方に値が入った時点で「提出完了」となること
- [ ] 上長が修正依頼フラグを立てると社員に通知が届くこと

### 自己評価入力（F-08）

- [ ] 評価フェーズ外では自己評価を提出できず、APIが `403 Forbidden` を返すこと
- [ ] `goal_sets.is_midterm_closed = TRUE` のフラグがある場合に限り、フェーズ外でも提出できること

### 上長評価入力（F-09）

- [ ] 社員の自己評価が未提出の場合、上長評価の提出ができないこと
- [ ] 上長評価スコアが自己評価スコアと異なる場合、コメント入力が必須となること

### 評価調整・確定（F-10）

- [ ] MBOスコアと360度評価スコア（成果・クレド）が仕様通りに合算されること
- [ ] 最終評価確定後に社員・上長が結果を閲覧できること

### 通知・リマインダー（F-13）

- [ ] 以下のイベントでメール通知が送信されること
  - 目標の承認申請 → 直属上長に通知
  - 目標の承認 → 申請者に通知
  - 目標の差し戻し → 申請者に通知（コメント本文を含む）
  - 修正依頼フラグ → 対象社員に通知
  - 最終承認後差し戻し（`MEETING_REJECTED`） → 申請者・直属上長に通知
  - フェーズ切替 → 全対象者に通知

### エクスポート（F-15）

- [ ] HR・ADMIN のみがエクスポート操作を実行できること
- [ ] 目標データ・評価スコアをCSV形式で出力できること
- [ ] 出力ファイルのエンコーディングはUTF-8（BOM付き）とすること

### イレギュラー社員対応（F-20）

- [ ] 期中目標設定・期中クローズはHR/ADMINのフラグ付与時のみ許可され、一般ロールは `403 Forbidden` を返すこと
- [ ] 期中昇格（等級2→3）時に旧 goal_set が論理削除（`is_active = FALSE`）され、新しい goal_set が作成されること

### SmartHR CSV連携（F-21）

- [ ] 更新対象フィールドは等級・雇用形態・1次評価者・2次評価者に限定されること
- [ ] バリデーションエラーが1件でもある場合、全件ロールバックされること
- [ ] 処理結果（成功件数・エラー件数・エラー行）が画面に表示されること

## 8. 機能要件

> **Section 7（受け入れ条件）との役割分担**  
> **Section 7**: 「何が満たされれば完了とみなすか」のテスト可能なチェックリスト  
> **Section 8**: 「どのようなルールで動作するか」の仕様・実装方針

### 8.1 認証・アクセス制御（F-01）

- Supabase Auth を使用したメールOTP認証（パスワード不要・2ステップ式）
- ロールベースのアクセス制御（ADMIN / HR / MANAGER / TEAM_LEADER / MEMBER）
- 1ユーザーが複数ロールを持てる
- アクセス制御はAPIレベルで実施する（フロントエンドの非表示のみでは不十分）

### 8.2 目標の公開範囲設定（F-05・F-11）

| 公開範囲 | 閲覧できる対象 |
|----------|----------------|
| `SELF_ONLY` | 本人と上長・HR・ADMINのみ |
| `DEPARTMENT` | 同一部署の全メンバー（デフォルト） |
| `COMPANY` | 全社員 |

### 8.3 評価スコア計算（F-10）

**MBOスコア計算:**

```
MBOスコア = Σ ( goal[i].score × goal[i].weight / 100 )
            for i in {KPI_1, KPI_2, ORG_CONTRIBUTION}
```

**360度スコア加算:**

- 成果スコアが4.5以上かつ全体回答者の上位20%以内 → +10ポイント
- 等級5以上でクレドスコア6.5以上 / 等級3〜4でクレドスコア6.0以上 → +3ポイント

### 8.4 通知・リマインダー（F-13）

通知イベントの一覧は Section 7（F-13 受け入れ条件）を参照。

- **通知チャネル**: メール（Supabase Edge Functions + AWS SES）
- **送信タイミング**: イベント発生時にリアルタイム送信（バッチ処理ではない）
- **メール本文**: 操作内容・対象者名・現在のステータス・システムへのリンクを含む
- **リマインダー**: フェーズ終了3日前に、未完了の対象者へ自動送信

### 8.5 監査ログ（F-16）

- 重要操作（目標設定・承認・評価確定・ユーザー操作等）のログを記録する
- ログ保持期間: 10年（目標・評価データと同等）
- ログに記録する情報: 操作日時・操作者・操作種別・対象レコードID・変更前後の値

### 8.6 エクスポート（F-15）

- **操作権限**: HR・ADMIN のみ実行可能
- **出力対象**: 目標データ（タイトル・詳細・スコア・重み）、評価スコア（自己・上長・最終）
- **フォーマット**: CSV（UTF-8 BOM付き）
- **絞り込み条件**: 評価期・部署・等級でフィルタリングして出力可能

### 8.7 SmartHR CSV連携（F-21）

- **更新対象フィールド**: 等級・雇用形態・1次評価者ID・2次評価者ID
- **処理方式**: 差分更新（既存レコードを上書き。レコードの削除は行わない）
- **エラー処理**: バリデーションエラーが1件でもある場合、全件ロールバック
- **CSVフォーマット**: 1行目はヘッダー行。エンコーディングはUTF-8（BOM付き）
- **処理結果**: 成功件数・エラー件数・エラー行番号とエラー内容を画面に表示

## 9. 非機能要件

### 9.1 パフォーマンス

| 指標 | 目標値 | 補足 |
| -- | --- | -- |
| 画面表示速度（初期ロード） | 3秒以内 | 通常時 |
| API応答時間（通常操作） | 500ms以内 | 通常時 |
| 同時接続ユーザー数 | 100名 | ピーク時（評価期末・目標設定期）を想定 |

### 9.2 可用性

| 指標 | 目標値 |
|------|--------|
| 稼働率 | 99.5%以上（月次） |
| メンテナンス停止 | 月1回以内・深夜帯に実施 |

### 9.3 セキュリティ

- 全通信をHTTPS化する
- SQLインジェクション・XSS等のOWASP Top 10対策を実施する
- ロールによるアクセス制御をAPIレベルで徹底する（フロントエンドの非表示のみでは不十分）
- 個人情報（評価データ等）は適切なアクセス権限でのみ閲覧可能とする
- パスワードは使用しない。OTPはSupabase Authが生成・送信し、システム側でコードを保持しない

### 9.4 スケーラビリティ

- 現在80名・将来300名規模への拡張を前提とした設計とする
- ベータ版はVercel + Supabase構成で運用し、将来的なAWS移行を考慮してベンダー依存の実装を最小化する

### 9.5 データ保持

| データ種別 | 保持期間 |
|-----------|---------|
| 目標・評価データ | 10年 |
| 監査ログ | 10年 |
| セッションデータ | ログアウトまで |

### 9.6 バックアップ・障害対応

| 項目 | 方針 |
| -- | -- |
| バックアップ頻度 | 日次（Supabase自動バックアップ） |
| RPO（目標復旧時点） | 24時間以内 |
| RTO（目標復旧時間） | 2時間以内 |
| 障害時の対応フロー | 4（連絡先・エスカレーション手順） |

### 9.7 ブラウザ・デバイス対応

| 項目 | 方針 |
| -- | -- |
| 対応ブラウザ | Chrome・Safari・Edge（各最新2バージョン） |
| IE / 旧Edge | 非対応 |
| モバイル対応 | レスポンシブデザイン対応（PC優先。スマートフォンは閲覧のみ） |

### 9.8 コンプライアンス

- **個人情報保護法**: 評価データは個人情報として管理し、利用目的（社内評価）の範囲内でのみ使用する
- **データ主体の権利**: 社員は自身の目標・評価データをシステム内で閲覧できる
- **第三者提供の禁止**: 評価データは社外に提供しない
- **データ処理の委託**: Vercel・Supabase・SendGridへのデータ処理委託に際し、各社の利用規約・データ処理規約を確認する

### 9.9 テスト要件

- **評価スコア計算ロジック（F-10）**: ユニットテストのカバレッジ100%を目標とする
- **主要フロー（目標設定〜評価確定）**: E2Eテストで正常系・異常系をカバーする
- **承認フロー**: 4段階の各ステップ・差し戻し・再申請の動作をテストする
- **テスト環境**: 本番とは独立したステージング環境を用意し、リリース前に検証する
