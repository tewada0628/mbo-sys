## 3. 機能要件

### 3.1 認証（F-01）

- Supabase Auth を使用したID・パスワード認証
- ログイン後は JWT を用いたセッション管理
- 未認証状態でのアクセスはログイン画面へリダイレクト（Next.js middleware で制御）
- パスワードリセットはメールリンク方式（Supabase Auth 標準機能を使用）

### 3.2 目標設定・承認フロー（F-05）

- 目標セットは KPI連動目標2件（`KPI_1`/`KPI_2`）+ 組織貢献目標1件（`ORG_CONTRIBUTION`）= 計3件で構成
- 各目標項目には連動パターン・達成基準（1.2/1.0/0.8水準）・ウェイト・公開範囲を設定する
- ウェイト合計は必ず100%（APIバリデーション + DBトリガーで二重担保）。デフォルト: KPI_1=50%/KPI_2=30%/ORG_CONTRIBUTION=20%
- 公開範囲は `SELF_ONLY` / `DEPARTMENT`（デフォルト）/ `COMPANY` から選択
- **等級3以上の正社員（`is_mbo_target = TRUE`）**: 目標設定後に4段階承認フロー（本人→上長→事業部長→経営）を経て `APPROVED` に遷移
- **等級1〜2・契約社員・アシスタント（`is_mbo_target = FALSE`）**: 承認フロー対象外。保存後は即 `SAVED` ステータス。承認申請ボタンは画面に表示しない
- `APPROVED` および `PENDING_*` 状態では社員による目標内容の自由編集を禁止（APIレベルで403）。`DRAFT` / `REJECTED` / `MEETING_REJECTED` のみ編集可能
- **フェーズ制御**: 目標設定（新規）は `GOAL_SETTING` フェーズ内のみ許可。一般ロールがフェーズ外で操作した場合は 403 を返す。HR/ADMIN はフェーズ外でも操作可能

### 3.3 目標修正申請（F-06）

- 修正申請は `APPROVED` 状態の目標セットに対してのみ可能
- 修正理由コード（`KPI_CHANGE` / `STANDARD_DEVIATION` / `ROLE_CHANGE` / `MIDTERM_ENTRY` / `EARLY_CLOSURE` / `GRADE_PROMOTION`）の選択が必須
- 申請中の `goal_sets.status` は `APPROVED` を維持する
- 承認されると新バージョンの目標として保存し、変更前の内容は履歴として保持（`goals.is_current` フラグで管理）
- **社員起点**: 社員が修正案を作成して申請 → 4段階承認フローを再実施
- **上長起点**: 上長が中間振り返り時に修正依頼フラグ（`midterm_reviews.revision_requested`）を立てる → 社員に通知 → 社員が修正申請を起票

### 3.4 中間振り返り（F-07）

- `MIDTERM` フェーズ内のみ入力可能（フェーズ外は403）
- 社員コメントと上長振り返りコメントを各目標項目ごとに並列入力
- 上長コメントは `midterm_reviews.manager_comment` に保存
- 中間振り返りの完了条件: `employee_submitted_at` と `manager_submitted_at` が両方 NOT NULL
- 上長修正依頼フラグ（`midterm_reviews.revision_requested`）: 上長が立てると社員に通知が届き、F-06の修正申請を起票する起点となる

### 3.5 評価（F-08 / F-09 / F-10）

- **フェーズ制御**: 自己評価は `SELF_REVIEW` フェーズ、上長評価は `MANAGER_REVIEW` フェーズのみ入力可能
- 評価スコア: 1.2 / 1.0 / 0.8（共通）、および 1.5 / 2.0（上長評価のみ）
- 自己評価提出後にのみ上長評価が入力可能
- 自己評価と上長評価のスコアに差異がある場合は上長評価コメントの記入が必須
- **バイアス警告（F-18）**: 上長評価入力画面の上部にバイアス五大症状のガイダンスバナーを**静的に**表示する。スコアパターンの自動検知は行わない
- **評価調整・確定（F-10）**: HR/ADMIN が `is_mbo_target = TRUE` の社員のMBOスコアを確認・調整し、360度スコアを合算した上で総合評価（S/A/B/C/D）を確定する

### 3.6 イレギュラー対応（F-20）

- **期中目標設定**: 中途入社・復職者の入社/復職タイミングでの目標設定。HR/ADMIN が許可フラグを付与した場合のみ `GOAL_SETTING` フェーズ外でも新規作成可能（修正理由コード: `MIDTERM_ENTRY`）
- **期中クローズ**: 休職予定者（実勤務6か月以上）・退職者の期末前評価完了。HR/ADMIN が操作（修正理由コード: `EARLY_CLOSURE`）
- **期中昇格（等級2→3）**: 2月昇格時に旧 goal_set を論理削除し、`is_mbo_target = TRUE` の新 goal_set を作成して承認フローを開始。`target_months = 8`（2〜9月）を記録（修正理由コード: `GRADE_PROMOTION`）
- **評価対象外**: 実勤務期間6か月未満の場合、HR が `is_evaluation_exempt = TRUE` を設定。評価スコアの算出・昇降給判定から除外

### 3.7 通知（F-13）

- **P0スコープ（承認フローに統合）**: 承認・差し戻し時に `notifications` テーブルへのレコード生成を各 Route Handler 内で実施
- **P1スコープ**: 通知一覧画面（S-17）・Header の通知アイコン（未読バッジ）・メール通知（Supabase Edge Functions + SendGrid）

### 3.8 UIカラー方針

- ベースカラー: `#01AEBB`（プライマリ）+ `#FFFFFF`（白）
- 赤（`#c0392b`）: 差し戻し・エラー・危険のみ
- 緑・オレンジの使用禁止（プライマリカラーまたはグレーで代替）

## 4. 非機能要件

| 項目 | 要件 |
| -- | -- |
| レスポンスタイム | ページ初期表示 2秒以内（P90）、API応答 500ms以内（P95） |
| 同時接続 | 最大100名（ベータ版） |
| 可用性 | 99.5%以上（Vercel + Supabase の SLA 範囲内） |
| RBAC | 全 Route Handler で Supabase Auth による認証 + `lib/permissions.ts` によるロール・所有者チェックを先頭で実施。権限外の操作は 403 を返す |
| フェーズ制御 | フェーズ外操作は一般ロールに 403 を返す。HR/ADMIN は例外として全フェーズで操作可能 |
| 入力バリデーション | フロント（Zod + React Hook Form）と Route Handler（Zod）の両方で実施 |
| セキュリティ | XSS対策（React デフォルト）、SQLインジェクション対策（Prisma ORM）、CSRF対策（Next.js デフォルト） |
| 監査ログ | 承認・差し戻し・評価確定操作を `audit_logs` に記録。保持期間 10年 |
| ブラウザ対応 | Chrome / Edge / Safari の最新2バージョン |

## 5. 受け入れ条件

### P0 機能の受け入れ条件

#### 認証

- [ ] ログイン・ログアウトが動作すること
- [ ] 未認証アクセスがログイン画面へリダイレクトされること
- [ ] パスワードリセットメールが送信されること

#### 目標設定

- [ ] 等級3以上の社員が目標セットを作成・保存し、承認申請を送信できること（3件構成・ウェイト合計100%バリデーション）
- [ ] 等級1〜2・契約社員は目標を保存でき（`SAVED` 状態）、承認申請ボタンが表示されないこと
- [ ] 連動パターン（5種類）・達成基準・公開範囲（3種類）が設定できること
- [ ] `APPROVED` および `PENDING_*` 状態での PATCH が 403 を返すこと
- [ ] `REJECTED` 状態では編集ボタンが表示され、修正・再申請できること
- [ ] 目標設定フェーズ外での新規作成が一般ロールで 403 を返すこと

#### 承認フロー

- [ ] S-15 で4段階（上長→事業部長→経営）の承認・差し戻しができること
- [ ] 差し戻し後に社員が修正して再申請できること

#### 修正申請

- [ ] `APPROVED` 状態の目標セットで修正申請が送信でき、承認フローが再実施されること
- [ ] 申請中も `goal_sets.status` が `APPROVED` を維持すること
- [ ] 修正承認後、旧バージョンが `is_current = FALSE` として履歴に残ること

#### 中間振り返り

- [ ] 社員と上長がそれぞれコメントを独立して入力・保存できること
- [ ] 両者提出済みで「完了」と判定されること
- [ ] 上長修正依頼フラグを立てると社員に通知が届くこと

#### 評価

- [ ] S-06 で自己評価を入力・提出できること
- [ ] 自己評価提出前は S-07 の入力フォームがロックされていること
- [ ] S-07 でバイアス警告バナーが表示されること
- [ ] S-08 で `is_mbo_target = TRUE` の社員のスコアを確認・調整・総合評価を確定できること

#### 最終承認後差し戻し

- [ ] `DEPT_MANAGER` 以上（および HR/ADMIN）が `APPROVED` 状態の目標を `MEETING_REJECTED` に差し戻せること
- [ ] `UNIT_MANAGER` が操作した場合は 403 を返すこと

#### 全般

- [ ] 権限外の操作（ロール不一致・他者リソースへのアクセス）は 403 を返すこと

### P1 機能の受け入れ条件

- [ ] S-12/S-13/S-14 でマスタの CRUD 操作ができること

## 6. 制約事項

- パッケージマネージャー: **Bun**（`bun run dev` / `bun add`）
- ランタイム: Node.js LTS（Vercel/Supabase の互換性要件に従う）
- TypeScript strict mode 必須
- コンポーネントは shadcn/ui を優先使用（カスタムコンポーネントは最小限）
- DB 操作は必ず Prisma ORM 経由（生 SQL はマイグレーション・DBトリガーのみ）
- `any` 型使用禁止（`unknown` + 型ガードを使用）
- コメントは自明でない理由がある場合のみ1行で記述
